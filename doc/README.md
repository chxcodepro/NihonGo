# doc2 文档总览

## 一、文档目的

`doc2` 是 NihonGo 项目的主文档目录，用来统一记录项目规划、架构设计、开发规范、数据结构、安全要求、部署方式和版本策略。

这套文档主要服务 3 类场景：

1. **新人入项**：快速知道项目做什么、怎么分层、先看什么。
2. **开发实现**：减少口头同步，避免前后端、接口、数据结构理解不一致。
3. **上线运维**：让部署、监控、排障、发布回滚有据可查。

文档目标不是追求“写全所有理论”，而是保证下面几件事能落地：

- 需求边界清楚
- 模块职责清楚
- 实现约定统一
- 发布流程明确
- 出问题能快速定位

## 二、阅读顺序

### 2.1 新成员阅读顺序

建议第一次进入项目时按下面顺序阅读：

1. `PROJECT.md`：先理解项目目标、功能范围、技术栈、开发阶段。
2. `architecture.md`：再看整体架构、前后端分层、认证和部署拓扑。
3. `frontend-guide.md` / `backend-guide.md`：按自己的角色继续看开发规范。
4. `database-design.md`：理解核心数据模型和 Redis 结构。
5. `api-spec.md`：确认接口边界和统一响应格式。
6. `security-spec.md`：补齐安全边界，避免实现跑偏。
7. `deployment-guide.md`：了解部署、备份、CI/CD 和运维要求。

## 三、文档地图

### 3.1 项目总览类

- `PROJECT.md`：项目规划、目标、功能模块、技术栈、阶段计划。
- `architecture.md`：系统架构、数据流、认证流程、部署拓扑。

这一类文档回答的是：

- 项目为什么做
- 项目准备怎么做
- 各模块之间怎么协作

### 3.2 前后端开发类

- `frontend-guide.md`：前端目录规范、状态管理、组件和接口调用约定。
- `backend-guide.md`：后端项目结构、分层模式、响应规范、核心实现建议。
- `dev-setup.md`：本地环境搭建、依赖安装、常用命令和故障排查。

这一类文档主要解决：

- 本地怎么启动
- 代码应该放哪
- 功能应该怎么组织

### 3.3 数据与接口类

- `database-design.md`：数据库表结构、Redis 数据结构、迁移和备份策略。
- `api-spec.md`：接口路径、请求参数、响应结构、错误码和鉴权要求。
- `question-bank-spec.md`：题库目录、字段定义、类型约束和质量要求。
- `content-registry-spec.md`：题库和游戏内容的注册、索引、命名和校验规则。

这一类文档主要解决：

- 数据怎么存
- 服务之间怎么对接
- 静态题库和游戏内容如何组织、注册与校验

### 3.4 产品体验类

- `ui-design-spec.md`：颜色、字体、布局、组件、动效和响应式规则。
- `game-design.md`：打字竞速、推箱子语法游戏的规则和实现重点。
- `i18n-guide.md`：多语言资源结构、翻译流程和接入方式。

这一类文档主要解决：

- 页面长什么样
- 用户交互怎么玩
- 多语言如何统一管理

### 3.5 发布与安全类

- `security-spec.md`：认证安全、数据安全、接口安全和上线前检查。
- `deployment-guide.md`：Docker、Nginx、CI/CD、备份、资源要求。
- `version-update.md`：Web / Desktop / Mobile 更新策略和发版流程。
- `observability.md`：日志、指标、告警、巡检和排障流程。

这一类文档主要解决：

- 怎么安全上线
- 上线后怎么看系统状态
- 版本怎么更新和回滚

### 3.6 质量保障类

- `testing-strategy.md`：测试范围、测试分层、验收清单和质量门槛。

这一类文档主要解决：

- 什么必须测
- 什么程度才能发
- 出问题时如何尽早拦住

### 3.7 协作与维护类

- `contributing.md`：协作规范、分支策略、提交和评审要求。
- `seed-data.md`：开发测试数据、初始化数据、导入重置流程。
- `glossary.md`：统一术语，减少不同文档和模块之间叫法不一致。

这一类文档主要解决：

- 多人协作怎么配合
- 新环境怎么快速拉起数据
- 常见概念到底该怎么叫

## 四、各文档说明

### 4.1 `PROJECT.md`

项目总规划文档，负责定义产品目标、核心功能、技术栈、目录规划和阶段计划。第一次了解项目时优先看它。

### 4.2 `architecture.md`

系统架构说明文档，重点描述前后端分层、认证流程、数据流和部署拓扑。实现前用它确认模块边界。

### 4.3 `frontend-guide.md`

前端开发规范，覆盖 Monorepo 组织、页面结构、组件规范、状态管理、接口请求、主题和动效。前端实现前必读。

### 4.4 `backend-guide.md`

后端开发规范，覆盖 Go 项目结构、分层模式、统一响应、OAuth2、AI 代理、SRS 算法和测试建议。后端实现前必读。

### 4.5 `database-design.md`

数据库与缓存设计文档，记录核心表结构、字段语义、索引、Redis key 组织方式以及迁移备份策略。

### 4.6 `api-spec.md`

接口契约文档，规定请求方式、鉴权要求、统一响应格式、模块接口和错误码。前后端联调时优先对齐它。

### 4.7 `question-bank-spec.md`

题库数据规范，规定假名、词汇、语法、打字素材、测验题和游戏关卡的目录与字段结构。

### 4.8 `content-registry-spec.md`

内容注册规范，规定 manifest、pack、游戏内容注册、命名和校验规则。

### 4.9 `security-spec.md`

安全规范文档，定义密码、Token、OAuth2、限流、敏感数据处理和上线前安全检查项。

### 4.10 `deployment-guide.md`

部署指南，覆盖 Docker Compose、Nginx、CI/CD、日志查看、备份恢复和服务器要求。

### 4.11 `dev-setup.md`

开发环境搭建指南，帮助开发者快速拉起本地环境，并提供常见故障排查方法。

### 4.12 `ui-design-spec.md`

UI/UX 设计规范，用来统一视觉风格、页面布局、组件表现、动效节奏和无障碍要求。

### 4.13 `game-design.md`

游戏设计文档，描述小游戏目标、玩法流程、评分规则、关卡组织和交互重点。

### 4.14 `i18n-guide.md`

国际化指南，规定语言范围、资源目录、翻译工作流和集成方式，避免多语言实现分散失控。

### 4.15 `version-update.md`

版本更新策略，说明各端的版本检查、升级、发版和回滚方案。

### 4.16 `testing-strategy.md`

测试策略文档，规定测试范围、测试分层、发布前验收清单、覆盖率目标和缺陷分级。

### 4.17 `observability.md`

可观测性文档，规定日志字段、核心监控指标、告警规则、排障流程和日常巡检项。

### 4.18 `contributing.md`

协作与贡献指南，规定分支策略、提交规范、PR 要求、评审重点和文档同步规则。

### 4.19 `seed-data.md`

种子数据与初始化指南，说明基础数据范围、环境隔离、初始化流程、重置策略和验证清单。

### 4.20 `glossary.md`

术语表，用来统一项目中的业务术语、数据术语、运维术语和推荐叫法。

## 六、文档维护规则

### 6.1 必须同步文档的改动

下面这些改动默认必须同步文档：

- 新增或删除核心模块
- 目录结构调整
- API 路径、参数、响应结构变化
- 数据表结构变化
- Redis key 结构变化
- 安全策略变化
- 部署流程变化
- 版本更新流程变化
- 题库格式变化

### 6.2 更新原则

1. **设计先行**：能先改文档就先改文档，再落实现。
2. **同变更同步**：如果实现已经开始，文档最晚需要跟同一次提交或同一个任务同步。
3. **只保留当前真相**：废弃方案不要长期和现行方案并存，避免误导。
4. **例子要可用**：文档中的目录、命令、配置示例应尽量和当前项目结构一致。

### 6.3 文档粒度原则

- 总览写在总览文档，不把细节全堆进去。
- 模块细节写到对应模块文档，避免多个文件维护同一份内容。
- 接口、表结构、配置项尽量有唯一主文档。
- 同一个问题如果在多个文档都要提到，其他文档只写摘要并指向主文档。

### 6.4 命名规则

- 文件名统一使用小写中划线风格。
- 文档标题统一用中文。
- “规范”“指南”“设计”“策略”这几类命名尽量统一，不要同类文件混用不同叫法。

### 6.5 维护责任建议

- 产品范围变化：优先更新 `PROJECT.md`
- 架构调整：优先更新 `architecture.md`
- 前端实现变化：优先更新 `frontend-guide.md`
- 后端实现变化：优先更新 `backend-guide.md`
- 数据结构变化：优先更新 `database-design.md`
- 接口变化：优先更新 `api-spec.md`
- 上线和运维变化：优先更新 `deployment-guide.md`、`observability.md`

## 八、后续建议补充的文档

建议后续继续补：

- ADR / 决策记录目录：把关键方案从 `PROJECT.md` 中逐步拆开
- OpenAPI 或 Swagger 导出文件：便于接口联调和自动化校验
- 管理端或运营后台文档：如果后面增加后台能力，需要单独整理
- 题库编辑工作流文档：如果后续引入内容编辑人员，建议补单独流程说明

补充优先级建议：

1. 先补接口自动化与决策记录
2. 再补内容生产与运营相关文档
3. 最后补长期维护类配套文档

## 九、维护建议

为了避免文档和实现越来越偏，建议执行下面的最低规则：

- 每次新增功能前，先确认要改哪份主文档
- 每次联调前，先确认 `api-spec.md` 和 `database-design.md` 还是最新的
- 每次发版前，至少过一遍 `testing-strategy.md`、`deployment-guide.md`、`observability.md`
- 每个阶段收尾时，补一次“实现与文档是否一致”的检查

这份 `README.md` 的作用就是作为 `doc2` 的入口页。后续如果文档数量继续增长，建议优先维护这份文件，让新成员始终能快速找到路径。
