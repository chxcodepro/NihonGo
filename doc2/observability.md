# 可观测性文档

## 一、目标

可观测性的目标不是单纯“收日志”，而是让系统在上线后具备下面 4 种能力：

1. **及时发现故障**：服务挂了、接口错了、依赖异常了，要尽快看到。
2. **快速定位问题**：能从告警一路追到具体接口、具体请求、具体日志。
3. **衡量系统健康度**：知道当前系统是稳定、变慢，还是已经在风险边缘。
4. **支撑发布与回滚决策**：发版后如果错误率飙升，要能快速判断是否需要回滚。

## 二、观测范围

### 2.1 应用运行状态

需要持续关注：

- 服务是否存活
- 进程是否频繁重启
- 容器状态是否正常
- 健康检查是否通过
- 服务启动是否完成初始化

### 2.2 接口调用情况

需要持续关注：

- 请求量
- 成功率
- 错误率
- 平均响应时间
- P95 / P99 延迟
- 超时次数
- 慢接口排行

### 2.3 数据库与 Redis

需要持续关注：

- PostgreSQL 连接池使用情况
- 慢查询数量
- 死锁和事务冲突
- Redis 连接数
- Redis 命中率
- Redis 内存使用情况
- Redis key 增长趋势

### 2.4 AI 服务调用

需要持续关注：

- AI 请求量
- 成功率与失败率
- 上游模型服务可用性
- 首字返回时间
- 总响应时间
- 限流触发次数
- 单用户异常高频调用

### 2.5 前端关键异常

需要持续关注：

- 页面白屏
- 运行时脚本错误
- 接口失败提示数量
- SSE 断连率
- 版本不一致导致的资源加载错误

### 2.6 部署与版本更新

需要持续关注：

- 部署成功率
- 部署后错误率变化
- 新版本检查成功率
- 更新流程成功率
- 回滚触发次数

## 三、可观测性设计原则

### 3.1 先覆盖核心链路

优先接入可观测性的链路：

- 登录
- 学习内容拉取
- 学习进度保存
- 打字记录提交
- AI 对话
- 游戏记录提交
- 版本检查

### 3.2 日志、指标、告警要串起来

理想状态不是某个地方“有数据”，而是：

- 告警能提示出问题
- 指标能说明问题范围
- 日志能定位具体原因

### 3.3 优先记录能帮助排障的信息

不要只记“接口失败了”，还要尽量让日志回答这些问题：

- 哪个接口失败
- 哪个用户受影响
- 哪个环境出的错
- 请求走了多久
- 失败发生在哪一层

### 3.4 敏感信息绝不裸奔

可观测性要服务排障，但不能为了排障把敏感数据直接打出去。密码、Token、API Key、完整隐私文本都不能直接进入普通日志。

## 四、日志规范

### 4.1 日志分级

建议统一分为 4 级：

- `DEBUG`：开发调试信息，生产环境默认关闭或限量保留
- `INFO`：正常业务事件，如请求完成、任务执行完成
- `WARN`：可恢复异常或风险信号，如限流触发、外部依赖短暂波动
- `ERROR`：请求失败、依赖失败、系统异常

### 4.2 日志字段规范

建议后端结构化日志统一包含以下字段：

- 时间戳
- 日志级别
- 服务名
- 环境
- 请求 ID
- 用户 ID（有则记录）
- 接口路径
- 请求方法
- 状态码
- 耗时
- 错误码
- 错误摘要

这样做的好处是方便后续：

- 按请求定位问题
- 按用户定位影响范围
- 按接口看错误集中点

### 4.3 请求链路标识

建议统一引入请求链路标识：

- 每个请求分配唯一 `request_id`
- 前端发起请求时可以透传
- 后端日志必须带 `request_id`
- AI 调用、数据库异常、Redis 异常也尽量挂同一链路标识

这样出问题时，能够从一次接口调用一路追到内部依赖。

### 4.4 敏感信息脱敏

下面这些信息默认不能直接写入日志：

- 密码
- 完整 Token
- API Key
- 数据库密码
- 邮件验证码原文
- 用户隐私内容原文（按场景脱敏）

建议做法：

- 邮箱只保留部分可识别片段
- Token 只保留前后少量片段
- AI 对话内容按需要做摘要或脱敏，不全量直出

### 4.5 前后端日志边界

建议边界如下：

- 前端记录：页面错误、渲染异常、交互失败、网络异常、版本不一致
- 后端记录：业务执行、鉴权失败、依赖失败、数据读写异常
- 不要前后端把同一条业务信息重复打满两遍

## 五、指标监控

### 5.1 服务基础指标

建议监控：

- CPU 使用率
- 内存使用率
- 磁盘使用率
- 容器重启次数
- 进程存活情况
- goroutine 数量

### 5.2 接口指标

建议监控：

- 总请求数
- 各接口 QPS
- 2xx / 4xx / 5xx 分布
- 平均响应时间
- P95 / P99 延迟
- 超时次数
- 失败率

关键接口建议单独观察：

- `/api/auth/login`
- `/api/auth/refresh`
- `/api/learn/progress`
- `/api/typing/record`
- `/api/ai/chat`
- `/api/game/sokoban/record`
- `/api/version/check`

### 5.3 数据库指标

建议监控：

- 连接池使用率
- 活跃连接数
- 慢查询数量
- 查询平均耗时
- 死锁次数
- 备份执行状态

### 5.4 Redis 指标

建议监控：

- 内存使用量
- 命中率
- 连接数
- 过期 key 数量
- 被驱逐 key 数量
- 关键前缀 key 的增长趋势

### 5.5 AI 模块指标

建议监控：

- AI 请求数
- 成功率
- 失败率
- 超时率
- 首字返回时间
- 总响应耗时
- 限流命中次数
- 单用户峰值调用量

### 5.6 业务指标

建议保留基础业务指标，帮助判断发布后是否出现“功能没挂但用户用不了”的情况：

- 日活用户数
- 新注册用户数
- 学习完成次数
- 打字练习次数
- AI 对话次数
- 游戏完成次数
- 版本检查次数
- 更新成功率

## 六、健康检查

### 6.1 存活检查

存活检查只判断服务进程是否还活着，不做复杂依赖校验。它的作用是让容器平台知道“服务是不是已经死了”。

### 6.2 就绪检查

就绪检查用来判断服务是否已经具备接流量能力，建议检查：

- 配置是否加载完成
- 数据库是否可连接
- Redis 是否可连接
- 必要资源是否初始化完成

### 6.3 依赖检查

建议额外输出依赖状态，方便快速判断故障范围：

- PostgreSQL 状态
- Redis 状态
- AI 上游服务状态（可选）
- 题库资源是否可读
- 版本信息源是否可访问

## 七、告警策略

### 7.1 服务不可用告警

建议触发条件：

- 健康检查连续失败
- 容器频繁重启
- 服务进程退出

处理优先级应最高，因为这类问题直接影响整体可用性。

### 7.2 错误率告警

建议触发条件：

- 关键接口 5xx 错误率连续一段时间超过阈值
- 某类错误码突然暴涨
- 登录、学习进度保存、AI 对话、游戏记录等关键接口失败明显增加

### 7.3 延迟告警

建议触发条件：

- 关键接口 P95 明显升高
- AI 首字返回时间明显升高
- 数据库慢查询数量异常增加

### 7.4 资源使用告警

建议触发条件：

- CPU 长时间高占用
- 内存持续上涨并接近阈值
- 磁盘空间不足
- Redis 内存接近上限

### 7.5 AI 调用异常告警

建议触发条件：

- 上游模型服务错误率升高
- AI 请求超时明显增加
- 限流次数突然异常增长
- 单用户调用行为异常

### 7.6 数据库连接异常告警

建议触发条件：

- 连接池耗尽
- 慢查询持续升高
- 死锁增加
- 备份失败

## 八、排障流程

### 8.1 接口报错排查

建议按下面顺序排：

1. 看告警是否集中在某几个接口
2. 看错误率和延迟是否一起升高
3. 根据 `request_id` 查后端日志
4. 判断是参数问题、鉴权问题、依赖问题还是代码异常
5. 必要时看数据库和 Redis 指标

### 8.2 登录失败排查

优先检查：

- 数据库用户查询是否正常
- Redis 中验证码和 OAuth state 是否存在
- JWT 配置是否正确
- OAuth2 回调地址是否一致
- 最近是否改过认证相关配置

### 8.3 AI 对话异常排查

优先检查：

- 上游 AI 服务是否可达
- 限流是否触发
- SSE 是否被代理层中断
- 超时配置是否过短
- 模型配置是否为空或无效

### 8.4 打字或游戏异常排查

优先检查：

- 前端运行时错误
- 题库或关卡 JSON 是否缺字段
- 成绩计算逻辑是否异常
- 提交记录接口是否失败

### 8.5 部署后故障排查

优先检查：

- 最新版本是否包含配置变更
- 数据迁移是否成功
- 环境变量是否缺失
- 构建产物是否完整
- 错误是否在发版后立即上升

如果问题影响主链路且短时间难以定位，应优先准备回滚。

## 九、日志与监控工具建议

### 9.1 日志

建议使用结构化日志，至少保证：

- 可按时间筛选
- 可按请求 ID 查询
- 可按级别过滤
- 可按接口和用户做聚合分析

### 9.2 指标

建议至少建立下面几类面板：

- 服务基础资源
- 接口成功率与延迟
- 数据库与 Redis 状态
- AI 调用情况
- 关键业务量趋势

### 9.3 前端异常

建议至少能看到：

- 页面白屏
- 运行时异常
- 资源加载失败
- 接口报错聚合
- 版本不一致问题

### 9.4 告警通知

建议分级通知：

- 高优先级：服务不可用、核心链路故障
- 中优先级：错误率升高、延迟明显升高
- 低优先级：资源接近阈值、趋势异常

避免告警过多导致“响了也没人看”。

## 十、上线前检查项

- [ ] 健康检查接口可用
- [ ] 关键接口有基础指标
- [ ] 错误率与延迟有可视化面板
- [ ] 请求日志已带请求 ID
- [ ] 敏感信息已脱敏
- [ ] 数据库与 Redis 已接入基础监控
- [ ] AI 调用异常有可见信号
- [ ] 部署后能快速定位版本与请求链路
- [ ] 告警通知链路已验证

## 十一、日常巡检清单

### 11.1 每日巡检

建议查看：

- 服务是否正常运行
- 前一天错误率是否异常
- AI 请求失败是否异常
- 数据库与 Redis 是否接近资源阈值
- 备份任务是否成功

### 11.2 每周巡检

建议查看：

- 慢接口排行
- 慢查询排行
- 限流命中情况
- 磁盘空间增长趋势
- 高频异常日志聚类

### 11.3 每次发布后巡检

建议重点看：

- 登录接口是否稳定
- 学习进度保存是否稳定
- AI 对话是否稳定
- 游戏记录提交是否稳定
- 版本检查是否正常
- 新增错误类型是否出现

## 十二、落地建议

项目早期可以先从最小可用版本开始，不需要一开始就上很重的整套体系，但至少要先做到：

1. 关键接口有结构化日志
2. 请求链路有唯一标识
3. 错误率和延迟有基础面板
4. 服务、数据库、Redis 有基础监控
5. 核心故障有基础告警

先把最容易影响发布和排障的能力建起来，再随着项目增长逐步补更细的 tracing、聚合分析和业务看板。
