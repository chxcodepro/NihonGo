# 种子数据与初始化指南

## 一、文档目的

这份文档用来说明项目的初始数据怎么准备、怎么导入、怎么重置、不同环境下哪些数据该有、哪些不该有。

它主要解决下面几个问题：

- 新环境启动后，最少需要哪些基础数据
- 开发和测试环境怎么快速造数据
- 题库和数据库数据的边界怎么划分
- 数据初始化怎么做到可重复执行
- 怎么避免测试数据误进生产

## 二、种子数据的定位

种子数据不是业务长期运行过程中产生的数据，而是**让系统具备“可启动、可联调、可验证”能力的基础数据**。

种子数据主要用于：

- 初始化系统基础配置
- 准备开发和测试环境
- 提供最小可用演示数据
- 保证 CI 或预发布环境能稳定拉起

不建议把下面这些内容混进种子数据：

- 生产真实用户数据
- 临时测试垃圾数据
- 只在本地一次性用的手工试验数据

## 三、环境划分

### 3.1 开发环境

开发环境允许存在：

- 演示账号
- 最小题库样本
- 本地测试用版本数据
- 调试用黑名单域名样本

开发环境目标是：**快速启动，快速验证。**

### 3.2 测试 / 预发布环境

测试环境建议包含：

- 更接近真实规模的题库
- 联调账号
- 完整版本信息样本
- 更接近生产的配置结构

测试环境目标是：**上线前尽可能模拟真实行为。**

### 3.3 生产环境

生产环境只允许初始化真正必要的基础数据，例如：

- 版本信息基础结构
- 临时邮箱黑名单
- 必要系统配置

生产环境默认**不应写入演示账号、调试数据、随手测试数据**。

## 四、数据边界

### 4.1 哪些属于种子数据

建议纳入种子数据范围的内容：

- 应用版本基础数据
- 临时邮箱黑名单基础数据
- 必要系统配置
- 开发测试账号（仅开发/测试环境）
- 最小联调用题库索引数据

### 4.2 哪些不属于种子数据

下面这些不建议当种子数据维护：

- 用户运行过程中产生的学习进度
- 打字历史记录
- AI 对话历史
- 游戏成绩
- 线上真实运营数据

这些数据应该由系统运行自然产生，不该在初始化脚本里硬塞。

### 4.3 题库与数据库的边界

本项目题库主数据以 `packages/question-bank/` 为准。

建议边界如下：

- **题库正文**：放在 `packages/question-bank/`
- **数据库**：保存用户数据、行为数据、版本数据、缓存索引、系统配置
- **题库相关辅助数据**：如后续有导入状态、版本映射、统计信息，可入库

不要把完整题库正文和数据库种子数据搅在一起维护。

## 五、推荐初始化内容

### 5.1 应用版本数据

建议初始化：

- 至少一个当前版本
- 每个平台的基础版本结构
- 最小版本号与是否强更字段

典型用途：

- 版本检查接口联调
- Web / Desktop / Mobile 更新逻辑验证

### 5.2 临时邮箱黑名单

建议初始化：

- 常见临时邮箱域名基础集
- 域名更新时间
- 是否启用状态

典型用途：

- 注册校验
- 安全策略验证

### 5.3 开发测试账号

仅开发或测试环境建议初始化：

- 普通用户
- 已完成邮箱验证的用户
- 已有学习进度的用户
- 已绑定 OAuth 的用户（如后续支持模拟）

注意：

- 密码必须是测试专用密码
- 不要使用真实邮箱
- 数据内容要可重建

### 5.4 最小题库样本

建议准备：

- 少量五十音样本
- 少量 N5 词汇样本
- 少量 N5 语法样本
- 少量打字素材
- 少量游戏关卡

典型用途：

- 本地快速联调
- 前后端基础功能验证
- CI 轻量校验

## 六、目录与组织建议

### 6.1 推荐目录

建议目录大致如下：

```text
server/
├── migrations/
├── seeds/
│   ├── README.md
│   ├── base/
│   │   ├── 001_app_versions.sql
│   │   ├── 002_disposable_domains.sql
│   │   └── 003_system_config.sql
│   ├── dev/
│   │   ├── 101_dev_users.sql
│   │   └── 102_dev_progress.sql
│   └── test/
│       ├── 201_test_users.sql
│       └── 202_test_versions.sql
```

### 6.2 文件组织原则

- 基础数据和环境专属数据分开
- 按执行顺序编号
- 文件名直接体现用途
- 同一类数据不要散落在多个目录

## 七、初始化原则

### 7.1 可重复执行

初始化脚本必须尽量支持重复执行。

建议做到：

- 重复执行不会产生重复主数据
- 重复执行不会破坏已有合法数据
- 对冲突数据要么跳过，要么更新到目标状态

### 7.2 环境隔离

必须区分：

- 开发环境数据
- 测试环境数据
- 生产环境数据

禁止把开发演示账号、测试密码、临时样本数据直接推进生产。

### 7.3 可追踪

建议种子数据变更具备可追踪性：

- 谁改的
- 改了什么
- 为什么改
- 影响哪个环境

### 7.4 最小必要原则

种子数据不要越堆越多，够用就行。

如果某条数据只为某次临时排查而存在，排查完就不应该继续长期保留。

## 八、初始化执行流程

### 8.1 新环境初始化建议流程

建议流程：

1. 执行数据库迁移
2. 导入基础种子数据
3. 导入环境专属种子数据
4. 检查关键表是否已有最小数据
5. 验证核心接口是否可用

### 8.2 本地开发初始化流程

推荐流程：

1. 清空旧测试数据库
2. 重跑迁移
3. 导入基础数据
4. 导入开发账号和最小样本数据
5. 启动前后端联调

### 8.3 CI / 测试环境初始化流程

推荐流程：

1. 建立干净数据库
2. 执行迁移
3. 导入轻量但完整的测试数据
4. 执行自动化测试
5. 测试完成后按需要销毁环境

## 九、数据重置策略

### 9.1 什么时候需要重置

常见场景：

- 本地数据被污染
- 联调状态混乱
- 测试结果互相影响
- 迁移脚本改动后需要重新验证

### 9.2 建议重置范围

根据场景选择：

- 只清用户行为数据
- 清数据库但保留题库文件
- 数据库和 Redis 一起清
- 全量重建开发环境

### 9.3 重置原则

- 先备份需要保留的数据
- 明确当前是在开发、测试还是生产环境
- 生产环境禁止随意执行“全清空”式操作

## 十、题库导入与种子数据配合

### 10.1 推荐做法

推荐把题库导入看成独立流程：

1. 题库文件放在 `packages/question-bank/`
2. 先通过结构校验
3. 再根据需要生成数据库辅助数据
4. 最后由前后端按统一格式消费

### 10.2 不推荐做法

不推荐：

- 在 SQL 种子文件里塞整份大题库
- 一边改 JSON，一边手工改数据库，最后不知道谁是真相
- 让前端和后端各自维护一套题库格式

## 十一、验证清单

导入或初始化完成后，建议至少检查：

- [ ] 数据迁移已执行成功
- [ ] 应用版本数据存在
- [ ] 临时邮箱黑名单基础数据存在
- [ ] 开发或测试账号可用
- [ ] 题库最小样本可被正常读取
- [ ] 登录主链路可用
- [ ] 学习内容可加载
- [ ] 版本检查接口可返回数据

## 十二、常见风险

### 12.1 重复插入

原因常见于：

- 没做唯一约束
- 种子脚本不支持幂等
- 环境多次执行同一脚本

### 12.2 环境串数据

原因常见于：

- 测试环境误连生产库
- 开发脚本使用了生产配置
- 没区分环境专属数据

### 12.3 题库与数据库不一致

原因常见于：

- 改了题库 JSON 没同步更新解析逻辑
- 改了数据库辅助结构没更新文档
- 没做导入后的校验

## 十三、后续建议

后续建议逐步补齐：

- 初始化脚本说明文件
- 数据重置脚本
- 开发账号约定
- 题库导入脚本
- 校验脚本

如果后面把脚本补起来，这份文档就作为总说明，脚本只负责执行，不再重复解释规则。

## 十四、一句话原则

种子数据的核心原则就是：

**基础数据最小化、环境数据隔离化、执行过程可重复、真实生产绝不混入测试垃圾。**
