# 测试策略文档

## 一、测试目标

本项目测试策略的目标不是“把所有地方都测一遍”，而是优先保证主链路稳定、关键风险可控、发布前问题能尽量提前暴露。

测试工作的核心目标如下：

1. **保证核心功能可用**：认证、学习、打字、AI、游戏、版本更新这些关键路径不能一碰就坏。
2. **保证关键链路稳定**：前后端、数据库、缓存、题库之间的配合要稳定。
3. **保证多端体验基本一致**：Web、Desktop、Mobile 至少在核心行为上保持一致。
4. **保证接口和数据契约不乱**：避免前后端、服务与题库、服务与数据库之间互相对不上。
5. **保证发版前有最低门槛**：不是功能写完就发，而是达到一套明确的验收标准再发。

## 二、测试范围

### 2.1 Web 前端

Web 端测试重点包括：

- 页面是否能正常访问和渲染
- 页面加载状态、错误状态、空状态是否正确
- 表单校验是否符合预期
- 关键组件交互是否正常
- 状态切换是否正确
- API 成功、失败、超时场景是否有合理反馈
- 国际化切换是否正常
- 主题切换是否正常
- 关键动画是否影响功能可用性

### 2.2 Go 后端

后端测试重点包括：

- Handler 层参数解析和参数校验
- Service 层核心业务逻辑
- Repository 层数据库与缓存读写
- 统一响应结构和错误码输出
- JWT、Refresh Token、OAuth2、邮箱验证码逻辑
- 限流、权限判断、异常处理
- AI 代理调用和流式输出逻辑

### 2.3 数据库与缓存

数据层测试重点包括：

- 数据迁移能否正常执行
- 核心表约束是否有效
- 索引是否满足关键查询需要
- Redis key 是否按约定写入与过期
- Token 黑白名单逻辑是否正确
- 缓存命中、失效、回源逻辑是否正确

### 2.4 题库数据

题库测试重点包括：

- JSON 格式是否合法
- 必填字段是否齐全
- 字段值是否在允许范围内
- 多语言字段是否完整
- 引用关系是否存在断链
- 样本数据是否能被前端和后端实际消费

### 2.5 多端能力

多端测试重点包括：

- Web、Desktop、Mobile 的核心流程是否一致
- 登录后学习数据同步行为是否一致
- 版本检查逻辑是否一致
- 本地存储和恢复逻辑是否符合平台预期
- 平台差异是否有兜底方案

### 2.6 安全能力

安全相关测试重点包括：

- 未登录访问受保护接口会被拒绝
- Token 过期和刷新逻辑正确
- OAuth2 state 校验有效
- 限流规则可触发
- 非法参数、恶意参数能被拦住
- 敏感信息不会泄露到接口响应和日志中

## 三、测试原则

### 3.1 风险优先

优先测试下面这些高风险区域：

- 认证相关
- 学习进度保存
- AI 对话链路
- 游戏记录提交
- 版本检查与更新
- 数据迁移和缓存失效

### 3.2 主链路优先

对于资源有限的阶段，优先保证下面几条主链路可用：

1. 注册 / 登录 / 登出
2. 拉取学习内容并保存进度
3. 完成一次打字练习并查看结果
4. 发起一次 AI 对话
5. 进入小游戏并提交记录
6. 检查版本并显示更新信息

### 3.3 自动化优先拦截重复问题

下面这些适合优先自动化：

- 纯逻辑模块
- 接口契约校验
- 数据格式校验
- 回归频率高的主流程
- 容易因重构被改坏的模块

### 3.4 手工测试保留体验类问题

下面这些问题仍然需要保留手工验证：

- 输入法和打字体验
- 动画卡顿
- UI 错位
- 多语言溢出
- 暗色模式可读性
- 跨端交互手感

## 四、测试分层

### 4.1 单元测试

#### 目标

验证函数、方法、纯逻辑模块本身的正确性。

#### 适用范围

- 工具函数
- 数据转换函数
- DTO 校验逻辑
- Token 生成和解析
- SM-2 间隔重复算法
- 限流规则计算
- 前端 store 更新逻辑
- 题库数据解析逻辑

#### 要求

- 正常路径要有
- 边界值要有
- 异常路径要有
- 尽量不依赖真实数据库和外部服务

### 4.2 集成测试

#### 目标

验证多个模块拼在一起时是否能正确协作。

#### 适用场景

- Handler + Service + Repository
- 数据库读写 + Redis 缓存
- 登录 + Token 刷新
- 学习进度保存 + 查询
- 游戏成绩提交 + 统计更新
- AI 会话创建 + 历史查询

#### 要求

- 可以依赖测试数据库和测试 Redis
- 测试结束后应能清理数据
- 优先覆盖主链路和高风险链路

### 4.3 接口测试

#### 目标

验证 API 契约是否稳定，避免联调时出现“文档对不上实现”的问题。

#### 关注点

- 路径、方法是否正确
- 鉴权要求是否正确
- 参数缺失时返回是否正确
- 成功响应结构是否一致
- 错误响应结构是否一致
- 分页字段是否统一
- 错误码是否稳定
- 限流触发时返回是否符合约定
- SSE 事件格式是否符合约定

### 4.4 端到端测试

#### 目标

模拟真实用户行为，验证从页面到接口再到数据落库的一整条流程。

#### 必测主链路

1. 新用户注册并登录
2. 浏览五十音并完成一次练习
3. 学习一个词汇并保存学习进度
4. 完成一次打字练习并查看结果
5. 发起一次 AI 对话并收到流式回复
6. 进入小游戏并提交成绩
7. 修改设置并确认重开后仍生效
8. 检查版本更新提示

#### 说明

- E2E 数量不求多，但必须覆盖关键路径
- E2E 主要用于防止“页面能打开但流程走不通”

### 4.5 手工验收测试

#### 目标

兜底那些自动化不容易发现、但用户一眼就能感知的问题。

#### 重点关注

- 页面布局错位
- 输入法兼容问题
- TTS 是否能正常播放
- 多语言文案是否溢出
- 深色模式是否可读
- 移动端点击区域是否正常
- Desktop 和 Web 的差异是否过大

## 五、各模块测试重点

### 5.1 认证模块

必须覆盖：

- 邮箱注册成功
- 重复邮箱注册失败
- 验证码错误失败
- 验证码过期失败
- 邮箱登录成功
- 密码错误失败
- Refresh Token 刷新成功
- 刷新后旧 Token 失效
- OAuth2 state 校验失败时拒绝登录
- 登出后受保护接口不可继续访问

### 5.2 学习模块

必须覆盖：

- 五十音数据可拉取
- 词汇、语法列表支持分级查询
- 学习进度可保存
- 学习进度可读取
- 复习列表生成逻辑正确
- 匿名学习与登录后同步逻辑符合预期

### 5.3 打字练习模块

必须覆盖：

- 打字内容加载正常
- 计时开始和结束逻辑正确
- 正确率、错误数、WPM 计算正确
- 输入法特殊情况不会大量误判
- 成绩记录提交成功
- 已登录和未登录状态处理正确

### 5.4 AI 对话模块

必须覆盖：

- 会话创建成功
- SSE 流式返回正常
- 超时和断流时有可理解提示
- 会话历史可查询
- 会话删除生效
- 配置错误时返回合理错误
- 限流触发时返回合理错误

### 5.5 游戏模块

必须覆盖：

- 打字竞速配置可读取
- 推箱子关卡可加载
- 成绩记录可提交
- 星级和评分逻辑正确
- 异常关卡数据不会导致页面直接崩溃

### 5.6 版本更新模块

必须覆盖：

- 能正确判断是否有新版本
- 无新版本时不误报
- 更新日志可读取
- Desktop 更新流程可执行
- 版本号比较逻辑正确

### 5.7 国际化模块

必须覆盖：

- 三种语言切换正常
- 缺失文案有兜底
- 动态变量替换正确
- 长文案不明显破版
- 日期和数字显示符合本地化预期

## 六、测试环境

### 6.1 本地开发环境

用于开发者日常自测，要求：

- PostgreSQL 和 Redis 可用
- 支持本地快速重置数据
- 可以使用简化数据集
- 第三方依赖可按需 mock

### 6.2 CI 环境

用于提交校验和主分支保护，建议至少执行：

- 代码格式检查
- 类型检查
- 后端单元测试
- 前端单元测试
- 题库数据校验
- 国际化 key 一致性校验
- 核心集成测试

### 6.3 预发布环境

用于上线前联调和回归，要求：

- 配置尽量接近生产
- 能验证数据库迁移
- 能验证部署结果
- 能跑主链路验收
- 能验证版本检查和基础监控

## 七、测试数据策略

### 7.1 测试账号

建议至少准备下面几类账号：

- 普通用户
- 新注册用户
- 有学习进度的老用户
- 绑定 OAuth 的用户
- 专门用于限流测试的用户

### 7.2 题库测试数据

建议准备 3 套数据：

1. **最小数据集**：适合本地开发快速验证
2. **标准数据集**：适合联调和常规测试
3. **异常数据集**：适合验证兜底、容错和数据校验逻辑

### 7.3 数据清理与重置

建议支持下面几种能力：

- 一键清空测试数据库
- 一键清空测试 Redis
- 恢复默认题库数据
- 重新初始化测试账号

## 九、质量门槛

### 9.1 覆盖率目标

建议采用稳妥目标：

- 后端核心业务模块覆盖率不低于 70%
- 工具函数和纯逻辑模块覆盖率不低于 80%
- 前端关键 store / hooks 覆盖率不低于 60%
- 至少有一套主链路 E2E 验证

### 9.2 阻塞发布的问题定义

下面这些问题默认阻塞发布：

- 登录主链路失败
- 学习进度无法保存
- AI 对话不可用
- 小游戏核心流程不可用
- 数据迁移失败
- 关键页面白屏或崩溃
- 严重安全问题

### 9.3 缺陷分级建议

- **P0**：系统不可用，必须立即处理
- **P1**：核心功能严重异常，阻塞发布
- **P2**：重要功能异常，但有规避手段
- **P3**：体验问题、边角问题或不影响主链路的问题

## 十、CI 中的测试执行顺序

建议执行顺序如下：

1. 安装依赖
2. 代码格式检查
3. 类型检查
4. 后端单元测试
5. 前端单元测试
6. 数据与文案校验
7. 集成测试
8. 构建验证
9. 主链路 E2E（按阶段接入）

这样安排的好处是：

- 先挡住低级错误
- 再验证逻辑正确
- 最后验证系统级流程

## 十一、常见失败场景与排查办法

### 11.1 数据库连不上

优先检查：

- PostgreSQL 是否启动
- 连接配置是否正确
- 端口是否被占用
- 测试库是否初始化完成

### 11.2 Redis 连接失败

优先检查：

- Redis 是否启动
- 地址与密码配置是否正确
- 旧测试数据是否污染结果
- key 前缀是否与其他环境冲突

### 11.3 OAuth2 登录失败

优先检查：

- 回调地址是否一致
- state 是否过期或未写入 Redis
- 第三方应用配置是否正确
- 测试环境是否误用了生产配置

### 11.4 AI 对话测试失败

优先检查：

- 上游模型服务是否可达
- 是否需要 mock
- 超时配置是否过短
- SSE 代理层是否截断连接

### 11.5 打字测试不稳定

优先检查：

- 浏览器输入法差异
- 键盘事件和输入事件触发时机
- 统计计算是否受异步状态影响
- 测试脚本是否过度依赖精确时序

## 十二、执行建议

项目早期不需要一口气把所有测试体系都铺满，但至少要先做到下面这些：

1. 纯逻辑模块有单元测试
2. 核心接口有契约测试
3. 主链路有基础 E2E
4. 发版前有固定验收清单
5. 数据和文案有校验脚本

这样可以先把最容易出事故的地方挡住，再随着项目迭代逐步补齐更细的测试。
