# 协作与贡献指南

## 一、文档目的

这份文档用来统一项目协作方式，减少多人开发时常见的问题：

- 需求理解不一致
- 分支命名混乱
- 提交信息看不懂
- PR 太大不好审
- 改了代码没改文档
- 改完功能没做最基本自检

目标很简单：**让开发、联调、评审、发版更顺。**

## 二、适用范围

这份指南适用于下面几类贡献：

- 新功能开发
- 缺陷修复
- 文档补充
- 题库更新
- 配置调整
- 测试补充
- 部署与运维脚本修改

无论是长期维护者，还是临时参与修改的贡献者，都建议先过一遍这份文档。

## 三、协作原则

### 3.1 先对齐，再动手

开始写代码前，至少要先对齐下面几件事：

- 这次要解决什么问题
- 改动范围在哪
- 会影响哪些模块
- 是否需要改接口、数据库、题库或文档

如果需求本身还不清楚，优先补问题定义，不要一上来就写代码。

### 3.2 小步提交，避免大爆炸

建议把改动拆成小步：

- 一次只解决一个相对完整的问题
- 功能、重构、文档尽量分开提交
- 单个 PR 尽量聚焦一个主题

这样做的好处是：

- 更容易 review
- 更容易回滚
- 更容易查问题

### 3.3 设计、实现、文档同步

下面这些改动，默认需要同步文档：

- 新增功能模块
- API 变化
- 数据结构变化
- 题库格式变化
- 部署方式变化
- 安全策略变化

如果只是代码改了、文档没跟，后面很容易越维护越乱。

### 3.4 先保证正确，再考虑炫技

项目优先级建议是：

1. 先保证功能正确
2. 再保证结构清楚
3. 再考虑性能和体验优化
4. 最后才是“看起来高级”的技巧

## 四、分支策略

### 4.1 主分支说明

建议分支职责保持清晰：

```text
main            → 生产分支，保护分支
develop         → 开发主分支
feature/*       → 功能分支
fix/*           → 修复分支
docs/*          → 文档分支
refactor/*      → 重构分支
release/*       → 发布分支
```

### 4.2 分支命名建议

命名尽量短、直接、可读：

```text
feature/kana-chart
feature/ai-chat-stream
fix/tts-no-sound
fix/login-refresh-token
docs/api-pagination
refactor/learn-service
release/1.0.0
```

### 4.3 什么时候开新分支

建议下面这些场景都单独开分支：

- 新功能
- 缺陷修复
- 较大文档补充
- 数据结构调整
- 部署与配置调整

不要长期直接在 `main` 或 `develop` 上堆改动。

## 五、提交流程

### 5.1 标准流程

建议流程：

1. 从 `develop` 拉最新代码
2. 新建功能分支或修复分支
3. 开发与自测
4. 同步文档
5. 提交代码
6. 发起 PR
7. 通过评审后合并

### 5.2 开发前检查

动手前先确认：

- 需求范围明确
- 是否影响接口
- 是否影响数据库
- 是否影响题库
- 是否影响多端行为
- 是否需要更新文档

### 5.3 提交前自检

提交前至少完成：

- 代码能正常运行或能说明为什么现在跑不了
- 相关测试已跑，或至少完成最小手工验证
- 没把敏感配置写死
- 没有明显无用代码、调试代码、临时日志
- 文档已同步

## 六、Commit 规范

### 6.1 基本格式

建议使用统一前缀：

```text
feat: 新功能
fix: 修复
docs: 文档
style: 格式调整
refactor: 重构
test: 测试
chore: 杂务
perf: 性能优化
```

### 6.2 推荐写法

推荐格式：

```text
type(scope): 简短说明
```

示例：

```text
feat(learn): 添加五十音图交互表格
fix(tts): 修复 Chrome 下日语发音失败
docs(api): 补充分页和错误码约定
refactor(auth): 拆分 token 刷新逻辑
test(ai): 补充 SSE 流式接口测试
```

### 6.3 提交说明原则

好的提交信息应该回答：

- 改了什么
- 改的是哪个范围
- 为什么改

尽量避免这种写法：

- `update`
- `fix bug`
- `修改一下`
- `临时提交`

## 七、Pull Request 规范

### 7.1 PR 标题建议

PR 标题尽量能直接说明主题，例如：

- 补充 AI 模块 SSE 事件协议
- 修复学习进度重复写入问题
- 新增题库数据校验流程文档

### 7.2 PR 描述建议包含

- 变更背景
- 变更范围
- 影响模块
- 是否涉及接口或数据库
- 是否涉及文档更新
- 验证方式
- 风险点

### 7.3 PR 大小建议

为了便于 review，建议：

- 小 PR 优先
- 单个 PR 尽量只处理一个主题
- 如果改动特别大，先拆设计 PR，再拆实现 PR

## 八、Code Review 关注点

### 8.1 功能正确性

重点看：

- 逻辑是否符合需求
- 边界条件是否考虑
- 异常路径是否处理
- 是否会引入重复写入、状态错乱、权限问题

### 8.2 结构清晰度

重点看：

- 模块职责是否清楚
- 是否有不必要的耦合
- 是否有明显重复代码
- 命名是否清楚

### 8.3 风险与兼容性

重点看：

- 是否影响现有接口
- 是否影响数据库结构
- 是否影响题库结构
- 是否影响 Web / Desktop / Mobile 一致性
- 是否影响安全边界

### 8.4 文档与测试

重点看：

- 是否补了对应文档
- 是否补了必要测试
- 如果没补，理由是否成立

## 九、文档同步规则

### 9.1 哪些改动必须同步文档

下面这些改动默认需要同步 `doc2`：

- 新增页面或模块
- 新增或修改接口
- 数据表或缓存结构变化
- 题库字段变化
- 安全规则变化
- 部署流程变化
- 发版和更新策略变化

### 9.2 文档同步优先级

建议优先更新：

1. `PROJECT.md`
2. `architecture.md`
3. 对应模块文档
4. `README.md`

如果来不及全部补完，至少要先把主文档补对。

## 十、测试与验证要求

### 10.1 最低验证要求

每次变更至少要提供一种验证方式：

- 自动化测试
- 构建通过
- 手工验证步骤

### 10.2 高风险改动要求

下面这些改动建议提高验证标准：

- 认证相关
- 数据库迁移
- AI 对话链路
- 学习进度写入
- 游戏成绩写入
- 版本更新逻辑

### 10.3 文档类改动

纯文档改动可以不跑构建，但建议至少检查：

- 文件结构是否正确
- 标题层级是否清楚
- 内容与现有文档是否冲突

## 十一、题库与数据变更要求

### 11.1 题库改动

修改题库时建议同步确认：

- 字段结构是否变了
- 是否要更新 `question-bank-spec.md`
- 是否影响前端渲染
- 是否影响后端解析
- 是否需要补校验脚本

### 11.2 种子数据改动

修改种子数据时建议同步确认：

- 是否区分开发、测试、生产环境
- 是否能重复执行
- 是否会污染生产环境
- 是否需要更新 `seed-data.md`

## 十二、紧急修复流程

### 12.1 适用场景

适用于：

- 生产故障
- 登录主链路中断
- 数据损坏风险
- 严重安全问题

### 12.2 处理原则

1. 先止血
2. 再补详细修复
3. 修完补回归验证
4. 最后补文档和复盘

### 12.3 紧急修复后补项

紧急修复结束后，建议补：

- 原因说明
- 影响范围
- 验证方式
- 是否需要补测试
- 是否需要补监控和告警

## 十三、沟通建议

### 13.1 需求不清时

优先问清楚：

- 目标是什么
- 边界是什么
- 不做什么
- 是否有现成方案或参考

### 13.2 出现争议时

优先讨论：

- 哪个方案更稳
- 哪个方案后续维护成本更低
- 哪个方案更符合当前项目阶段

不是谁声音大就用谁的方案。

## 十四、合并前检查清单

- [ ] 改动范围清楚
- [ ] 分支命名合理
- [ ] Commit 信息可读
- [ ] 相关文档已同步
- [ ] 已完成最小验证
- [ ] 无明显调试残留
- [ ] 无敏感信息硬编码
- [ ] PR 描述完整

## 十五、一句话原则

这个项目的协作原则可以概括成一句话：

**先对齐，后实现；小步提交，文档同步；改完自检，再发合并。**
